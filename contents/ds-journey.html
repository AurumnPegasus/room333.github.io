Finding shortest Euclidean distance between two points from a set of n points

An easier problem

This question was part of 6.046J / 18.410J Design and Analysis of Algorithms Spring 2015 Assignment 1. (Removed story part) (Distance referred here is Euclidean distance)

Given a set of nnn points on a 2D2D2D plane, find if any 222 points are within distance of 1 to each other.

We will try to solve this problem and then try to generalise it.

Let P={Set of given points sorted by their x-coordinate}P = \{\text{Set of given points sorted by their x-coordinate}\}P={Set of given points sorted by their x-coordinate}.

We will use a divide and conquer algorithm to solve this problem.

Divide:

if (len(P) > 2):
	Divide P into two halfs

Conquer:

if (len(P) <= 2):
	if (distance between points in P <= 1):
		return True
	else:
		return False
else:
	Solve each half

Merge:
Merging is the fun part of this algorithm. First notice that if either of the halfs return True, we do not need to solve the algorithm further as we have already found if such a pair exists.

if (solve(P_left) == True OR solve(P_right) == True):
	return True

So now, we know that all of the points on the left half are seperated by atleast distance 1 and same for the right half. So, we only care for [min(X,Y):X,Y∈Pleft,Pright][min(X, Y) : X, Y \in P_{left}, P_{right}][min(X,Y):X,Y∈Pleft​,Pright​]

Lets try the naive algorithm, pairing each point from XXX with YYY and calculating the distance between the. This should give us an O(n2)O(n^2)O(n2) merging step, analysing the total complexity of the algorithm:
T(n)=2T(n/2)+O(n2)T(n)=O(n2)T(n) = 2 T(n / 2) + O(n^2) \\ T(n) = O(n^2)T(n)=2T(n/2)+O(n2)T(n)=O(n2)
So, we did all this work with divide and conquer to achieve nothing better than simple brute force. But lets try to analyse the merging step more carefully.

The figure above shows the merging step. The black line divides the set into two halves. Now, lets make an important observation:

We only care about the points which lie in the are enclosed by the red lines.

It is obvious that any point outside the red strip area cannot be within distance 1 of a point from the other half. Lets sort these coordinates by their y-coordinate and put them in a list P′=[(x,y) ∣ ∣x−mid∣<=1]P' = [(x, y)\ |\ |x - mid| <= 1]P′=[(x,y) ∣ ∣x−mid∣<=1]. Now, I claim that for each point in P′P'P′ we only need to check distance from 7 other points to determine if any of them lie at a distance of 1 from the other.

Proof:
For a point p′p'p′ in P′P'P′, we only need to check for points whose y-coordinate is lower than this point. So, we only need to check points whose y-coordinate lies in the range [p′.y−1,p′.y][p'.y - 1, p'.y][p′.y−1,p′.y]. The figure below illustrates how we check this:

For a particular point, we check for points which lie in a rectangle of 1×21 \times 21×2. How many points can lie in such a rectangle? An upper-bounded analysis shows that it is at most 7.

So our merging step can be reduced to O(n)O(n)O(n) if we implement it careful enough. Overall complexity of algorithm:
T(n)=2T(n/2)+O(n)T(n)=O(nlog⁡(n))T(n) = 2T(n / 2) + O(n) \\ T(n) = O(n\log(n))T(n)=2T(n/2)+O(n)T(n)=O(nlog(n))

Generalising the algorithm

Now, we have an algorithm to check if any two points are withing distance ddd of each other. And we want an algorithm to find the minimum distance between any two points in PPP.

The easiest way to achieve this is probably taking the previous algorithm and binary searching on ddd giving us an algorithm which works in O(nlog⁡(n)×iter)O(n\log(n) \times iter)O(nlog(n)×iter) where iteriteriter is the number of iterations we do the binary search for. Higher the number of iterations, higher the precision.

But can we do better? Yes we can!
Lets modify our previous algorithm with some simple modifications to achieve this.

Divide:
Same
Conquer:

if (len(P) == 1):

The same conditions hold, we only need to check for 7 points and thus merging in O(n)O(n)O(n). So overall complexity:
